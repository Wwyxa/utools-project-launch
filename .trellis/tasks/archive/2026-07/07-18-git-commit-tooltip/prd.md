# 优化 Git 提交悬浮卡片

## Goal

提高 Git 提交树悬浮卡片的可读性，使作者与时间的元信息、提交标题、提交正文和变更摘要形成清晰且连贯的阅读顺序，并在数据可用时显示作者头像。

## Confirmed Facts

- 提交树和悬浮卡片由 `src/components/project/GitTab.vue` 控制。
- 当前 tooltip 将标题置顶、作者/时间置于标题下方，正文单独渲染，尚未显示变更摘要。
- `ProjectGitCommitSummary` 已提供提交哈希、标题、正文、作者、时间及可选的文件变更数据；单个文件变更包含新增、删除和状态。
- 逐提交文件变更已经通过 `readGitCommitFiles` 按需加载，现有提交展开区正在使用该能力；tooltip 可以复用同一数据源。
- 项目已经读取 Git remote URL，但提交模型目前没有作者邮箱或头像 URL。
- GitHub 官方的“按提交获取”接口可依据 GitHub remote 的所有者、仓库名和现有提交哈希返回关联作者的 `avatar_url`；无需读取或传输作者邮箱。
- 浏览器预览使用降级桥接，不能读取真实本地 Git 数据；真实数据由 `public/preload.js` 提供。

## Requirements

- 悬浮卡片优先提高文字层次和扫描性，避免元信息、标题和正文混在同一视觉层级。
- 作者和时间放在卡片顶部，标题与正文放在其下方并保持连续阅读。
- 增加接近 VS Code Git 视图的变更摘要：已改文件数量、新增行数和删除行数，并使用清晰的正负语义颜色。
- 在 GitHub remote 存在时，按需查询提交关联的 GitHub 作者头像；无法获得远程头像时必须有稳定、无网络依赖的回退显示。
- 头像请求必须缓存、不得阻塞 tooltip 的基础内容，也不得影响非 GitHub 仓库、离线环境或浏览器预览。
- 保持现有提交树的鼠标悬浮、键盘关闭、视口避让和文件预览选择行为。

## Acceptance Criteria

- [ ] 悬浮卡片从上至下依次呈现作者头像与作者/时间、提交标题与可选正文、变更摘要，且长文本不会遮挡、重叠或超出卡片。
- [ ] 元信息在紧凑宽度下可换行或截断，标题和正文在视觉上仍连续、可辨识。
- [ ] 变更摘要准确显示该提交的变更文件数、总新增行数和总删除行数；数据加载中和不可用时都有清晰且不误导的表现。
- [ ] GitHub remote 的公开提交在头像查询成功时显示 GitHub 作者头像；无法识别、网络不可用、未授权私有仓库或非 GitHub remote 时显示由作者信息生成的本地首字母头像。
- [ ] 同一提交的头像不会重复触发网络请求；头像加载或失败不会延迟 tooltip 的作者、时间、标题、正文和变更摘要显示。
- [ ] 真实 uTools 运行环境与浏览器预览均可正常工作，现有 Git 操作没有回归。

## Avatar Decision

- 仅对 GitHub remote 按需查询提交关联的 GitHub 头像，使用本地首字母作为全部失败场景的默认回退。
- 不使用 Gravatar，避免根据作者邮箱向第三方服务发送可关联的标识。
- 本次不增加 GitHub 登录、令牌管理、持久化头像缓存或用户可配置的头像提供方。

## Research Status

- 已完成本地实现调查。
- 已核验 VS Code 核心源码：Git 历史 provider 将提交哈希、作者名、作者邮箱和头像尺寸交给可注册的头像 provider，并将返回 URL 写入历史项的 `authorIcon`；无结果时回退为账号图标。
- 本次不复制 VS Code 的扩展注册层，而以 GitHub remote 与提交哈希直接取得头像，保留同样的非阻塞本地回退体验。

## Notes

- 已验证的数据流：现有提交标题、正文和文件变更数据足以支持卡片布局与变更摘要；头像增强仅需 GitHub remote 和现有提交哈希，无需扩展提交作者邮箱。
