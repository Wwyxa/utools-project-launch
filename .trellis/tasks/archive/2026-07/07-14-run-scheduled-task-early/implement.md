# 实施计划

## Ordered Checklist

- [x] 在 Pinia store 中新增按 `entryId` 提前执行计划条目的 action，并在调用现有执行核心前完成未来待执行、脚本和项目并发校验。
- [x] 在 `AutomationTab.vue` 增加提前执行处理函数、可用性判断和独立操作反馈。
- [x] 将当天未来待执行条目渲染为带 `Play` 图标按钮的紧凑控件，保证非待执行条目展示不变，并补齐无障碍名称。
- [x] 在中英文自动化文案中增加“提前执行”操作名称，复用现有开始/阻塞反馈文案。
- [x] 复查任务级“立即执行”调用路径未被改变。
- [x] 在 `Dashboard.vue` 的“下次执行”列表中加入提前执行处理函数和独立图标按钮，传递原 `nextEntry.id`。
- [x] 保留“下次执行”行的鼠标与键盘详情导航，并通过停止事件传播避免提前执行后跳转。

## Validation

1. 运行 `npm run type-check`，验证 Vue 模板、store action 和本地化键类型一致。
2. 运行 `npm run build`，验证生产构建。
3. 浏览器手动检查当天有多个未来计划条目时，每个条目均显示入口，布局在窄窗口不溢出。
4. 提前执行一个未来条目，确认该条目进入运行并最终结束，历史保留原计划时间，计划中没有新增 `manual` 条目。
5. 模拟同项目已有任务运行，确认操作被拒绝并显示反馈；确认停用定时执行后仍可提前执行。
6. 等待或调整到原计划时间，确认已提前执行的条目不会再次运行。
7. 在首页任务概览检查“下次执行”列表的提前执行按钮、反馈、详情导航和窄窗口布局。

## Risk And Rollback Points

- 风险集中在动作入口校验与计划条目状态竞态；必须由 store 作为最终校验边界，不能只依赖组件禁用状态。
- 不修改 `runAutomationTask` 的核心执行流程，避免影响正常到点执行、现有手动立即执行和恢复逻辑。
- 若界面布局回归，可单独回滚条目内按钮并保留 store action，不涉及数据恢复。
