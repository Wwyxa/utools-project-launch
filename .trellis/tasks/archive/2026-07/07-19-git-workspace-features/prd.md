# Git Repository Workspace Visibility

## Goal

Help developers understand and move between the Git repositories that make up a project without leaving the project launcher, with particular emphasis on `git worktree` and submodule relationships used by parallel AI-assisted development workflows.

## Background

- The current Git Tab already covers working-directory changes, staged changes, diffs, commits, local branch switching, detached-HEAD checkout, remotes, upstream status, and fetch/pull/push.
- In the current Git Tab implementation, "worktree" names the changed-file review state; it does not model linked worktrees created by `git worktree`.
- The current Git snapshot contract contains repository, status, branch, remote, and history data, but no linked-worktree or submodule records.
- Git commands are executed by the uTools preload bridge. Browser preview exposes the same typed bridge with unavailable or empty fallbacks.
- Existing Git Tab guidance requires a compact two-pane tool layout; this feature must not add a persistent third pane.
- No earlier session decision about linked worktrees or submodules was found in local Trellis conversation history.
- The first implementation used a Git top-bar trigger plus a teleported Worktrees/Submodules overview. User review found that interaction ineffective compared with VS Code Source Control because the related repositories remained detached from the working changes, commit controls, and history context.
- The supplied VS Code reference keeps a persistent repository tree above the source-control content. Selecting the main repository or a related repository changes the repository context consumed by the changes and history regions instead of opening a separate overview dialog.

## Requirements

- R1: Expose linked Git worktrees as structured data, including path, ref, HEAD, locked/prunable health, staged/unstaged/untracked-entry counts, and upstream ahead/behind counts. Count untracked directories as one entry through Git's normal untracked mode, label the value as "untracked entries" rather than a physical-file count, read per-worktree status with bounded concurrency, and isolate failures to the affected entry.
- R2: Expose direct submodules configured by the current repository as structured data, including path, configured URL and branch when present, recorded commit, checked-out commit, initialization state, branch or detached state, and change state. Do not recursively expand nested submodules in the first release.
- R3: Turn the existing Git Tab top repository-status bar into a compact, conditional repository section. A project with only its main repository omits the repository selector and redundant repository name entirely, leaving the existing branch/remote status actions. When any linked worktree or direct submodule exists, show one compact selector that combines a small expand chevron with the selected repository name; its expanded state lists the main repository and linked worktrees as peer repository rows, with direct submodules indented beneath their owning main repository. Unavailable, uninitialized, bare, or prunable entries remain visible but cannot become the active Git context. The first discovery of any related repository expands the section once, after which manual expand/collapse state is remembered for that project during the renderer session and ordinary refresh never overrides it. The section opens inline without a dialog, remains usable in the plugin's narrow window sizes, and clearly identifies the selected repository context only when that distinction is useful.
- R4: Selecting a healthy repository row must switch the Git Tab's repository context atomically: repository metadata, changed files, commit history, diff reads, expanded commit details, AI analysis context, and ordinary Git content actions must all target the selected repository. Ordinary actions include stage, unstage, discard, commit, branch/commit checkout, remote operations, and AI commit-message generation. Keep the launcher project identity unchanged and do not add related paths to the persisted project list.
- R5: Keep existing Git status, review, commit history, branch, remote, and AI flows available without behavioral regression.
- R6: Represent unsupported, missing, uninitialized, locked, prunable, or otherwise unhealthy entries explicitly instead of silently omitting them.
- R7: Preserve browser-preview compatibility through typed empty or unavailable fallbacks; do not simulate successful local Git operations.
- R8: Keep repository-relationship lifecycle management read-only: do not create, remove, prune, lock, initialize, update, sync, or deinitialize Git worktrees or submodules. Existing ordinary Git content actions are allowed against the selected healthy repository, but every store action must re-authorize the repository target against the latest workspace snapshot before reading or writing.
- R9: Replace the teleported repository-workspace overlay and its trigger with the expandable section built into the current Git Tab top bar, modeled after VS Code Source Control. The section controls both existing Git panes instead of occupying either pane. Selecting a repository must not leave stale file review, commit selection, expanded commit files, tooltips, commit-message drafts, or asynchronous results from the previous repository visible in the new context.
- R10: Remember the selected repository and the user's explicit repository-section expand/collapse choice separately for each launcher project during the current renderer session. Switching tabs or projects and returning restores those choices, while renderer reload starts from the main repository and lets R3's related-repository discovery rule choose the initial expansion. If the remembered target disappears or becomes unavailable, automatically fall back to the main repository. Never persist or synchronize a related repository's local absolute path or repository-section visual state.
- R11: Keep repository rows and the conditional selector compact and predictable. Clicking the row body selects the repository; every expanded row shows repository identity, branch or detached state, change count, and health. Hide the selector for main-only projects; when related repositories exist, combine a small expand chevron and the selected repository name in one control without a repository-kind subtitle or decorative glyphs. Preserve the familiar top-right fetch, pull, and push icon actions for the selected repository, but do not add a summary-row More menu. Editor, terminal, folder, and path-detail actions remain in each expanded row's More menu. The project-detail header keeps the only refresh button and, while Git is active, refreshes the selected repository plus the root workspace inventory. Unselectable entries expose path/status details but no Git actions. Branch and Remote menus render outside clipped Git panels and close on outside click or Escape.
- R12: Isolate commit-message drafts by repository context during the current renderer session. Switching repositories restores that repository's draft; renderer reload clears all drafts. While a Git write action is running, repository switching is disabled. AI commit-message generation may be superseded by a repository switch, but its result is generation-bound to the originating repository draft and can never overwrite the newly selected repository's input. Drafts and generations are not persisted or synchronized.

## Acceptance Criteria

- [ ] AC1: A repository with multiple linked worktrees shows the main repository and every linked worktree as peer rows, with each path, current branch or detached state, HEAD identifier, locked/prunable health, staged/unstaged/untracked-entry counts, and upstream ahead/behind counts when available. An untracked directory counts as one entry, and one unreadable worktree reports its own error without hiding other entries or becoming selectable. (R1, R3, R6)
- [ ] AC2: A repository with configured direct submodules shows each submodule indented beneath its owning main repository, with path, declared URL and optional branch, initialization state, parent-recorded commit, checked-out commit, branch or detached state when available, change counts, and whether the checkout differs from the recorded commit. Uninitialized or unavailable submodules remain visible but cannot become the active Git context. (R2, R3, R6)
- [ ] AC3: Selecting the main repository, an available linked worktree, or an initialized direct submodule updates repository metadata, changed files, commit history, diff reads, expanded details, AI context, and all ordinary Git content actions to that exact repository while keeping the current launcher project identity. (R4)
- [ ] AC4: Repositories without linked worktrees or submodules omit the repository selector, repository name, and empty repository list while retaining all existing Git Tab functions. (R3, R5)
- [ ] AC5: Missing, uninitialized, locked, prunable, and inaccessible entries remain visible with actionable status text; one failing entry does not hide healthy entries. (R6)
- [ ] AC6: Browser preview renders the view with empty fallback data and clearly reports unavailable local actions. (R7)
- [ ] AC7: No worktree or submodule lifecycle mutation is exposed. Stage, unstage, discard, commit, checkout, remote, and AI commit-message actions can target a selected healthy related repository, are re-authorized at the store boundary, and can never target a different repository than the visible selected context. (R8)
- [ ] AC8: Type checking, build validation, focused parsing tests, and existing Git validation scripts pass. (R5)
- [ ] AC9: The existing Git Tab top bar conditionally exposes a collapsible repository selector: a main-only project shows no selector, first discovery of related repositories reveals and expands the compact selector once, later refreshes preserve the user's per-project renderer-session choice, collapsing retains the selected repository cue and existing status actions, expanding reveals repository rows without a dialog or a third pane, and selecting a row updates both existing panes. Repository switching clears or isolates repository-specific review, selection, expansion, tooltip, draft, pagination, and in-flight request state so no stale result crosses repository boundaries. (R3, R9)
- [ ] AC10: Returning to a project within the same renderer session restores its last healthy selected repository; renderer reload or invalidation of that target falls back to the main repository, and persisted/exported project data contains no selected-related-repository path. (R10)
- [ ] AC11: Repository rows remain usable at host-like narrow widths: main-only projects do not repeat the project name, related-repository projects show one compact chevron-plus-name selector with no repository-kind subtitle or summary More menu, the top-right fetch/pull/push buttons operate on the visible selected repository, expanded-row external navigation and path details remain available from More, Branch/Remote menus are not clipped, and exactly one project-detail refresh button refreshes the active repository context. (R11)
- [ ] AC12: Commit-message drafts remain separate for main, linked-worktree, and submodule contexts within the renderer session; switching is blocked during Git writes; a late or superseded AI generation updates only its originating repository draft or is ignored; reload and persisted/exported project data contain no draft or generation state. (R12)

## Out of Scope

- Recursive repository or submodule visualization beyond direct configured submodules.
- Full replacement for dedicated Git desktop clients.
- Automatic AI-agent orchestration across worktrees.
- Remote submodule URL or credential management.
- Worktree create, remove, prune, lock, and unlock operations.
- Submodule init, update, sync, and deinit operations.
- Replacing the current launcher project's identity or persisted path with a related repository path.
- Automatically registering worktrees or submodules as persisted projects.
