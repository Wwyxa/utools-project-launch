# 修复后台任务成功后误报退出码 4294967295

## Goal

让后台自动化恢复按运行批次关联进程结果，确保脚本实际以 `code 0` 完成后重新打开插件仍显示完成，不被同一脚本其他运行的结果误报为失败。

## Confirmed Facts

- 提交 `d5a7d44` 已增加按项目和脚本缓存最近完成结果的机制，用于插件恢复时补全后台任务终态。
- 当前恢复逻辑将存在 `error` 或 `code !== 0` 的最近结果直接判定为失败。
- 用户已观察到任务预期工作已经完成，但打开插件后显示失败，原因文本为“脚本退出码 4294967295”。
- 该脚本执行完后自行结束，插件没有请求停止进程，因此不能使用 `stoppedByUser` 解释该终态。
- 本机最小实验及检索结果确认 Windows 下 Node.js 会将 `exit -1` 表示为 `4294967295`（`0xffffffff`）。该值是进程真实返回的非零状态，不是可安全忽略的成功哨兵。
- 用户提供的实际 Python 脚本最后正常输出“脚本执行结束。”，同一次前台进程事件记录为 `exited with code 0`。
- 自动化计划条目已经持久化外层 `runId`，但该值没有传入 `runCommand`、preload 进程元数据或完成结果缓存。
- 当前 `completedProcessResultsByScript` 仅以 `projectId + scriptId` 为键；手动运行、连续自动化运行或复用同一脚本的其他任务都会覆盖缓存。
- 孤儿运行恢复只用 `endedAt >= plannedAt` 筛选脚本最近结果，无法证明结果属于当前计划条目的 `runId`。
- 项目加载时会先计算并启动到期计划，再异步恢复旧运行，增加新批次覆盖旧批次恢复依据的机会。
- 7 月 10 日实际失败批次 `automation-run-1783645560045-bw0my4` 包含脚本 `run` 和 `meta`；`meta` 正常完成，`run` 命中已配置的输出退出条件后由插件停止，原始进程码为 `4294967295`。
- 实时自动化上下文通过 `stopRequestedByAutomationExit` 将该停止视为成功，但该语义只存在于 renderer 内存；后台恢复只能看到 preload 缓存中的原始非零退出码，因此误判失败。

## Requirements

- 自动化外层 `runId` 必须贯穿脚本启动、preload 进程元数据、完成结果缓存和恢复查询。
- 自动化恢复必须按 `projectId + scriptId + runId` 精确获取结果，不得回退到该脚本的最近一次结果。
- 手动脚本运行结果不得进入自动化批次结果索引，也不得覆盖自动化恢复依据。
- 项目加载必须先恢复持久化的运行中条目，再启动新的到期计划。
- 并发触发的运行状态恢复不得为同一计划条目重复写入历史。
- 不得全局把 `4294967295` 归一化为成功；真实非零退出码和显式进程错误仍保留失败语义。
- 输出匹配成功触发的自动化停止必须作为结构化终态写入对应 `automationRunId` 的完成结果，并在 renderer context 丢失后保持成功语义。
- 手动停止、超时停止和未命中输出退出条件的 `4294967295` 不得被标记为输出匹配成功。
- 修复应同时覆盖自动化计划状态、自动化历史状态及脚本结果状态的恢复，避免三者不一致。
- 不改变前台运行任务和用户主动停止任务的既有行为。

## Acceptance Criteria

- [x] 自动化批次 A 以 `code 0` 完成后，即使同一脚本的手动运行或批次 B 返回 `4294967295`，恢复 A 仍显示完成。
- [x] 自动化批次 A 返回非零退出码后，即使同一脚本的批次 B 成功，恢复 A 仍显示失败并保留 A 的退出码。
- [x] 两个自动化任务复用同一脚本、以及一个任务包含多个脚本时，结果均只按各自 `runId` 组合，不跨批次混用。
- [x] 没有精确批次结果的旧孤儿条目安全标记为跳过，不使用脚本最近结果猜测终态。
- [x] 项目加载恢复旧运行期间不会先启动新的到期批次。
- [x] 同时发生的 focus、visibility 和初始化恢复不会为同一计划条目生成重复历史。
- [x] 真实 `exit -1` 和其他非零退出码仍显示失败，不被误判为完成。
- [x] 显式 `error` 结果仍恢复为失败。
- [x] 自动化计划条目、历史记录和各脚本结果对同一次恢复给出一致状态。
- [x] 普通手动脚本显式返回非零退出码时，既有失败行为不变。
- [x] 项目现有类型检查、构建及相关兼容性验证通过。
- [x] 输出匹配成功触发停止后，即使 Windows 原始退出码为 `4294967295`，后台恢复仍将对应脚本和任务标记为完成。
- [x] 同样的 `4294967295` 若没有输出匹配成功标记，仍恢复为失败。
- [x] 手动停止和超时停止不会继承或伪造输出匹配成功标记。

## Out of Scope

- 重构自动化调度器或进程管理架构。
- 改变任务超时、用户停止或错过计划的产品语义。
- 修改用户提供的 Python 脚本或把特定非零退出码配置为成功。

## Notes

- 根因位于进程结果缺少自动化批次身份，而非 Python 脚本退出码传播。
- 自动化验证已执行真实 preload 与 Pinia store 代码；真实 uTools 宿主中的后台完成、关闭并重开流程仍建议人工烟测。
- 第二根因属于成功终态语义跨 renderer/preload 边界丢失；批次身份正确后仍需持久化“输出匹配成功停止”的原因。
